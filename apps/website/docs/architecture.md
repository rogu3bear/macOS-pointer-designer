# Architecture

## Overview
WindowDrop's website is a Leptos CSR application that ships as static assets for the public marketing, pricing, support, changelog, and download experience. An Axum runtime exists only for the optional server-backed web checkout and recovery flow when `SITE_LIFETIME_MODE=web`.

## Stack
- **Framework**: [Leptos](https://github.com/leptos-rs/leptos) (v0.6) in CSR (Client-Side Rendering) mode.
- **Build Tool**: [Trunk](https://trunkrs.dev/).
- **Routing**: `leptos_router`.
- **Styling**: Hand-written CSS using CSS variables (`styles/`).
- **Markdown**: `pulldown-cmark` for rendering content at runtime (compiled via `include_str!`).
- **Server**: Axum (`windowdrop-server`) for the optional `/checkout/lifetime` and `/api/web-license/*` flow in web-checkout mode.

## Directory Structure
- `site/src/`: Application source.
- `site/styles/`: Global and component CSS (base, layout, components, responsive).
- `site/assets/`: Static SVGs.
- `site/public/`: Root-level files (robots.txt, sitemap.xml).
- `dist/`: The output directory generated by `trunk build`.

## Responsive Design
Fully responsive from 320px to 4K (3840px). Breakpoints in `responsive.css`:
- 320px min (small phones)
- 768px (tablet)
- 1280px (desktop)
- 1920px (2K)
- 2560px (2K+)
- 3840px (4K)

## Deployment

**Canonical (production)**: Cloudflare Pages serves the public site, versioned downloads, and the root-level Pages Functions bundle on `windowdrop.pro`. The public site should stay in `SITE_LIFETIME_MODE=in-app` unless the Pages proxy has already been verified live against a healthy `WEB_CHECKOUT_ORIGIN`. Root-level deployment and hosting details live in `../../docs/hosting.md`.

**Optional web-checkout backend**: Axum still serves the private checkout origin for `/checkout/lifetime` and `/api/web-license/*` when `SITE_LIFETIME_MODE=web`. Do not switch the public site into web mode until the custom-domain proxy smoke tests pass for checkout and recovery.

**Alternative (static-only)**: The `dist/` folder is self-contained and can be deployed to any static host (Cloudflare Pages, Vercel, Netlify, GitHub Pages). Ensure the host is configured to rewrite 404s to `index.html` for client-side routing to work. Static hosting can serve the marketing site and download flow, but it cannot serve the server-backed web checkout, purchase verification, or email recovery endpoints.

## CTA System

The site uses a config-driven CTA (Call-to-Action) system to support different monetization phases without code changes.

### Configuration (`src/config.rs`)

```rust
pub enum CtaMode {
    TrialDownload,  // Direct download link
    EarlyAccess,    // Same as trial, different label
    NotifyMe,       // Email capture form (pre-launch)
    AppStoreLink,   // Mac App Store link
}
```

The active mode determines:
- Primary CTA button label
- Button behavior (link vs. email capture)
- Whether email capture section is shown

Pricing surfaces are copy-driven and support two explicit lifetime flows:
- Free tier: Finder support
- In-app mode: `$7.99` one-time lifetime unlock purchased and restored via StoreKit
- Web mode: `$7.99` Stripe checkout via `/checkout/lifetime`, then activation via signed token or `windowdrop://activate?...`

### Components

- `CtaPrimary`: Renders the primary CTA based on config mode
- `CtaSecondary`: Static secondary button (e.g., "How it works")
- `EmailCapture`: Email signup form for pre-launch phase

## Analytics

Minimal interest tracking via console logging (MVP approach).

### Events Tracked

| Event | Trigger |
|-------|---------|
| `cta_primary_clicked` | Primary CTA button clicked |
| `cta_secondary_clicked` | Secondary CTA button clicked |
| `animation_viewed` | Hero animation completed |
| `waitlist_submitted` | Email form submitted |

### Implementation (`src/analytics.rs`)

In development builds, events are logged to the browser console for debugging. Production builds do not emit analytics/telemetry. To integrate with an analytics provider, update the `track_event` function.

## Pre-launch vs Release Mode

When `trial_url` and `store_url` are `None` (NotifyMe mode), the Download page:
- Shows email capture as primary action
- Replaces "Alternative Downloads" and "Verify" with a "When Released" section
- Installation step 1 wording adapts ("When released, download...")

When download URLs are set, full download flow (DMG/ZIP, SHA-256 verify) is shown.
